## 一、项目概述

### 1.1 项目背景

本项目旨在构建一套基于 **MCP（Model Context Protocol）** 标准的个人财务管理服务。通过标准化的 MCP Tool 接口，任何兼容 MCP 协议的 AI 客户端（Claude Desktop、Cursor IDE 等）均可调用本服务，实现以自然语言驱动的财务管理能力。

### 1.2 核心价值主张

| 传统记账 App | 本 MCP 服务 |
|---|---|
| 需手动打开 App，逐字段填写 | 在 AI 对话中自然描述，自动录入 |
| 固定报表，无法灵活查询 | 用自然语言提问，动态生成分析 |
| 静态预算提醒 | AI 主动识别异常，提供个性化建议 |
| 孤立应用，无上下文理解 | 与 AI 工作流深度集成，理解上下文 |

### 1.3 目标用户（当前阶段）

- **主用户**：个人用户（单用户模式），具备基本财务管理意识
- **扩展方向**：家庭多成员共享（数据隔离），未来版本实现

### 1.4 MCP 客户端兼容目标

- ✅ Claude Desktop（Anthropic 官方客户端）
- ✅ Cursor IDE（内置 MCP 支持）
- 🔄 其他兼容 MCP 标准的客户端（通用兼容）

## 三、Tool 精简策略

### 3.1 精简原则

| 策略 | 说明 |
|---|---|
| 移除冗余查询入口 | `get_transaction` 合并进 `list_transactions`（增加 `id` 参数即可单条查询） |
| AI 自推断替代独立 Tool | `suggest_category` 删除：AI 客户端本身具备分类推理能力，无需单独 Tool |
| 操作语义合并 | `toggle_category_visibility` 合并进 `update_category`（`hidden` 字段） |
| 恢复操作统一入口 | `restore_transaction` 删除，归并到 `restore_deleted_record`（通用恢复） |
| 净资产查询内联 | `get_net_worth` 删除，净资产数据已包含在 `list_accounts` 响应中 |
| 日志查询统一 | `get_change_history` 合并进 `list_audit_logs`（增加 `resource_id` 参数） |
| 移除暂不启用功能 | `set_monthly_budget` + 储蓄目标三个 Tool（共 4 个）按需求移除 |

### 3.2 Tool 数量对比

| 分类管理 | 5 | 4 | -1（toggle 合并入 update） |
| 预算管理 | 6 | 2 | -4（月度总预算+储蓄目标移除） |
| 分析报表 | 7 | 7 | 不变 |
| 净资产 | 3 | 2 | -1（get_net_worth 移除） |
| 投资记录 | 5 | 5 | 不变 |
| 审计日志 | 3 | 2 | -1（get_change_history 合并） |
| 订阅管理 | 0 | 3 | 新增模块 |
| 财务分摊 | 0 | 3 | 新增模块 |
| 标签体系 | 0 | 1 | 新增（list_tags，返回完整维度+预设值+使用统计） |

## 四、功能模块设计

### 4.1 模块全览

```

├── 📁 账户管理模块       账户的增删改查、余额追踪

├── 📁 分类管理模块       二级分类体系、自定义分类、隐藏
├── 📁 预算管理模块       分类预算（无月度总预算）、预算预警
├── 📁 分析报表模块       月报/年报、趋势分析、预算对比、现金流
├── 📁 净资产模块         资产负债追踪、月度快照
```

### 4.2 账户管理模块

**功能说明**：管理用户的所有资金账户（银行卡、支付宝、微信、现金等）

| 功能点 | 描述 |
|---|---|
| 新增账户 | 支持多种账户类型：活期/支付宝/微信/信用卡/投资账户/现金 |
| 编辑账户 | 修改账户名称、备注 |
| 删除账户 | 软删除（保留历史记录完整性） |
| 查询账户列表 | 返回所有账户及当前余额（实时计算），响应中包含净资产汇总 |
| 余额校正 | 支持手动校正账户余额（与银行账单对账） |

### 4.3 交易记录模块（核心模块）

**功能说明**：记录所有收入和支出明细，支持自然语言驱动录入

| 功能点 | 描述 |
|---|---|
| 单条记账 | 结构化字段输入（金额/分类/账户/时间/备注/标签），支持精确到秒的时间戳 |
| 批量自然语言录入 | 一段文字识别多条交易，保留原始文本，AI 自动分类并打标签 |
| 编辑记录 | 修改已有交易记录的任意字段 |
| 删除记录 | 软删除，保留审计痕迹 |
| 查询记录 | 支持多条件过滤（时间范围/分类/账户/金额区间/关键词/标签），支持单条 ID 查询 |
| 转账记录 | 账户间转账（不计入收支，仅影响账户余额） |

**时间格式**：交易时间支持精确到秒，格式 `YYYY-MM-DD HH:mm:ss`，默认为当前时间。

**批量自然语言录入示例**：

```
用户输入："今天早上喝了一杯瑞幸咖啡18元，中午点了外卖38元，下午打了个滴滴12元到公司"

→ 解析为 3 条交易：
  1. amount: 18, type: expense, category: 餐饮/咖啡, tags: ["咖啡"], date: today 08:xx
  2. amount: 38, type: expense, category: 餐饮/外卖, tags: ["外卖"], date: today 12:xx
  3. amount: 12, type: expense, category: 交通/网约车, tags: ["滴滴"], date: today 14:xx

→ source_text: "今天早上喝了一杯瑞幸咖啡18元，中午点了外卖38元，下午打了个滴滴12元到公司"（原始文本保留）
```

**AI 自动打标签逻辑**：

标签体系为 **5 个维度 × 各维度最多 1 个值** 的结构化标签，覆盖面越全越好。Server 端根据规则在响应中返回 `suggested_tag_labels`，AI 客户端决策后写入 `tag_labels`。

| 维度 | 预设值 | Server 端推断规则 |
|---|---|---|
| **消费方式** `method` | 线上 | 关键词匹配：淘宝/京东/拼多多/美团/饿了么/线上/App/扫码 |
| | 线下 | 关键词匹配：超市/便利店/门店/收银/POS；或非外卖的餐饮堂食 |
| | 外卖 | 关键词匹配：外卖/饿了么/美团外卖/配送/叮咚 |
| **消费行为** `behavior` | 日常消费 | 属于通勤/基本餐饮/超市/水电煤等高频规律类别 |
| | 冲动消费 | 属于购物/娱乐类别 且 金额 > 该分类近 30 天均值的 150% |
| **消费类型** `consumption_type` | 必选消费 | 一级分类为：住房/交通/医疗/餐饮（基本餐） |
| | 可选消费 | 一级分类为：娱乐/购物/个人护理/旅行 |
| **金额规模** `scale` | 大额支出 | 金额 ≥ 500 元（阈值可通过系统配置调整） |
| | 小额支出 | 金额 < 500 元 |
| **消费目的** `purpose` | 生存必需 | 一级分类为：住房/医疗/餐饮（早/午/晚餐）/通讯 |
| | 发展提升 | 一级分类为：教育/健身/通讯（宽带/专业工具） |
| | 享受休闲 | 一级分类为：娱乐/旅行/外出就餐/个人护理 |

> **注意**：`transfer`（转账）类型的交易不参与标签体系；`income`（收入）类型当前版本也不参与标签推断。

### 4.4 分类管理模块

**功能说明**：管理支出/收入的二级分类体系

**分类体系规则**：
- 最多支持 **2 级**：一级分类（父）→ 二级分类（子）
- 二级分类不可再创建子分类
- 预设分类不可删除，仅可隐藏
- 预算设置可绑定一级或二级分类
- 一级分类含 `group_label` 字段，用于界面分组展示（如「必要生活」「健康成长」），不参与分类节点逻辑

**🚫 不计入统计的分类（`count_in_stats = false`）**：

| 分类 | 推荐记法 | 说明 |
|---|---|---|
| 债务管理（信用卡还款 / 花呗还款 / 借贷往来） | 优先用 `transfer` 转账类型 | 账户间资金转移，天然不计入收支 |
| 资产购置（投资性黄金 / 收藏品） | 优先用 `transfer` 转账类型 | 现金转换为资产形态 |
| 往来收入（收还款 / 收垫付款 / 人情回礼） | `income` + `count_in_stats=false` | 偶发性，不值得建账户，贴标记即可 |

### 4.5 预算管理模块

**功能说明**：按分类设置月度预算，支持预警和滚动延续

**设计变更（v0.2）**：
- ❌ 移除月度总预算（`set_monthly_budget`）
- ❌ 移除储蓄目标功能
- ✅ 仅保留分类维度的预算设置

| 功能点 | 描述 |
|---|---|
| 设置分类预算 | 为指定分类设置月度预算，支持 `recurring`（自动延续到下月） |
| 查询预算状态 | 返回当前月所有分类的预算执行情况（已用/剩余/超支预警） |

**预警机制**：
- 每次调用 `record_transaction` 写入后，自动检查分类预算使用率
- 使用率 ≥ `alert_threshold`（默认 80%）时，响应中附带 `budget_warning`
- 使用率 > 100% 时标记为 `exceeded`

### 4.6 分析报表模块

| 报表类型 | 描述 | 支持维度 |
|---|---|---|
| 月度收支汇总 | 指定月份的收入/支出/结余概览，可选附带预算执行对比 | 按月/按账户 |
| 年度收支汇总 | 全年收支总览，月度对比 | 按年/按分类 |
| 分类支出趋势 | 指定分类的历史支出趋势 | 同比/环比 |
| 预算执行报告 | 预算 vs 实际支出对比，AI 文字建议 | 按分类/按月 |
| 消费习惯分析 | 消费频率、时间段分布、高频场景 | 按天/按时段 |
| Top N 支出 | 指定期间内支出最多的 N 条记录 | 可排序/过滤 |
| 现金流分析 | 按时间段展示资金流入/流出，识别资金高峰低谷 | 日/周/月 |

> 分析报表中的月度数据，自动将「财务分摊」计划中当月应分摊的金额纳入计算，而非计入大额支出发生月。

### 4.9 订阅管理模块（新增）

**功能说明**：追踪个人所有周期性订阅扣费，提供到期/扣费提醒

| 功能点 | 描述 |
|---|---|
| 新增订阅 | 记录订阅名称、金额、周期、扣费账户、分类、下次扣费日 |
| 编辑/取消订阅 | 修改订阅信息，或将状态设为「已取消」 |
| 查询订阅列表 | 返回所有订阅，高亮展示 7 天内即将扣费的订阅 |

**订阅周期支持**：monthly（月付）、quarterly（季付）、yearly（年付）、weekly（周付）

**典型订阅示例**：Netflix、Spotify、Apple One、ChatGPT Plus、腾讯视频、京东 Plus 等

### 4.10 财务分摊模块（新增）

**功能说明**：将一次性大额支出按月摊销，使月度报告更真实反映消费水平

**适用场景**：
- 房租一次性季付（如 9000 元付 3 个月 → 每月 3000）
- 年度保险费（如 2400 元/年 → 每月 200）
- 大件购物（如 6000 元笔记本 → 摊至 12 个月 → 每月 500）

**分摊逻辑**：
1. 用户记录大额支出时，同时创建分摊计划（总金额、起始月、摊销月数）
2. 分摊计划生效后，月度报告中该支出以「分摊月金额」计算，而非全额计入支出月
3. 原始交易记录保留完整金额，用于账户余额准确计算
4. 分摊计划可随时修改或取消（剩余月份恢复为实际支出）

| 功能点 | 描述 |
|---|---|
| 创建分摊计划 | 绑定一笔已有交易，设置起始月和总摊销月数 |
| 编辑/取消分摊 | 修改摊销参数或终止分摊计划 |
| 查询分摊列表 | 查看所有分摊计划及每月摊销进度 |

### 4.11 审计日志模块

| 功能点 | 描述 |
|---|---|
| 查询操作日志 | 查看所有 Tool 调用记录，支持按资源 ID 过滤（获取某条记录的完整修改历史） |
| 恢复已删除记录 | 从软删除中恢复任意类型的记录（交易/账户/分类等） |

## 五、非功能性需求

### 5.1 输入验证与安全

- **所有 Tool 入参**必须通过 Zod Schema 校验，拒绝无效输入
- **金额字段**：正数，最多保留 2 位小数，单笔上限 999,999,999
- **字符串字段**：最大长度限制，防止注入攻击
- **日期时间字段**：格式 `YYYY-MM-DD HH:mm:ss`，拒绝超过当前时间 1 年后的日期
- **SQL 注入防护**：通过 Drizzle ORM 参数化查询，禁止字符串拼接 SQL
- **审计日志**：所有写操作（增/改/删）均记录到 `audit_logs` 表

### 5.2 性能要求

- 单次 Tool 调用响应时间 < 500ms（常规查询）
- 支持分页查询，默认每页 50 条，最大 200 条
- 复杂分析查询（如年度报表）< 2000ms
- 数据库连接池：最小 2，最大 10 连接

### 5.3 错误处理

- 所有 Tool 返回统一的错误响应格式
- 数据库连接失败时返回友好错误信息，不暴露连接串
- 业务校验失败（如无效账户 ID 等）返回明确的错误原因

### 5.4 数据完整性

- 软删除策略：重要数据（交易记录、预算）不物理删除
- 外键约束：账户删除前必须先处理或转移相关交易
- 金额精度：数据库使用 `NUMERIC(15,2)` 类型，避免浮点数精度问题

## 六、MCP Tool 命名规范

所有 Tool 命名遵循 `动词_名词` 规则，动词统一使用：

| 动词前缀 | 含义 |
|---|---|
| `record_` | 创建/新增记录 |
| `batch_record_` | 批量创建记录（自然语言输入） |
| `update_` | 更新/修改记录 |
| `delete_` | 删除记录（软删除） |
| `get_` | 查询单条记录 |
| `list_` | 查询列表 |
| `analyze_` | 分析/统计报表 |
| `set_` | 设置配置项 |
| `restore_` | 恢复软删除记录 |

完整 Tool 清单详见 [02-MCP-Tools规格.md](./02-MCP-Tools规格.md)
